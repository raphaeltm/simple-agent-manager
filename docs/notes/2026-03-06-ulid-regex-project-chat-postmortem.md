# ULID Regex Rejection in Project Chat ACP WebSocket — Post-Mortem

## What Broke

Project chat could not establish ACP WebSocket connections to workspaces, preventing real-time per-token streaming and follow-up prompt delivery. Users saw messages only via the 3-second polling/DO WebSocket fallback, and follow-up prompts showed "Agent is not connected — message saved but prompt not delivered."

## Root Cause

The `useProjectAgentSession` hook (`apps/web/src/hooks/useProjectAgentSession.ts:33`) validated workspace IDs with a UUID-only regex:

```typescript
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
```

All workspace IDs in the system are ULIDs (26-char Crockford Base32), generated by `apps/api/src/lib/ulid.ts`. The regex silently returned `''` from `deriveWorkspaceWsHost()`, causing the ACP WebSocket to never connect.

Introduced in commit `b167624` (feat: unified ACP chat architecture with deploy fix).

## Timeline

- **2026-03-04**: `b167624` merged — unified ACP chat architecture introduced the UUID regex
- **2026-03-06**: Issue reported — production project chat messages not streaming. Investigation traced to ULID/UUID mismatch.

## Why It Wasn't Caught

1. **No test for the ID validation path**: The `deriveWorkspaceWsHost()` function was untested. A simple test with a real ULID would have caught this immediately.
2. **Silent failure mode**: The function returns `''` on validation failure instead of throwing. No console warning, no error banner. The fallback path (DO messages via polling) masked the failure.
3. **Dual-path architecture complexity**: The ACP WebSocket was one of two message paths. When it failed silently, the DO WebSocket + polling still delivered messages (with worse UX), making the bug non-obvious.
4. **No integration test for workspace ID format**: Component tests mocked `useProjectAgentSession` entirely, never exercising the real ID-to-URL derivation.

## Class of Bug

**ID format validation mismatch**: A validation regex assumed one ID format (UUID) while the system exclusively generates another (ULID). This class occurs when:
- Different parts of the system use different ID formats
- Validation is added without checking what the actual system produces
- Silent failures prevent detection

## Process Fix

Added `.claude/rules/` guidance: when adding input validation for internal IDs, verify the regex/validation against actual ID generation code. This is captured in the regression tests added in this PR:
- `apps/web/tests/unit/hooks/useProjectAgentSession.test.ts` — Tests VALID_ID_RE against both UUID and ULID formats
- `apps/web/tests/unit/components/project-message-view.test.tsx` — Tests the DO + ACP message merge logic
