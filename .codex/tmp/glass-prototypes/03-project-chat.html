I'll investigate the OAuth redirect loop on Safari mobile. Let me start by reading the auth callback handler to understand the current flow.

Read file apps/api/src/routes/auth.ts

I found the issue. The callback handler at handleOAuthCallback() is using document.cookie to set the session token after the OAuth exchange. Safari's Intelligent Tracking Prevention (ITP) blocks third-party cookies by default, which means the session cookie set during the redirect is being silently dropped.

When the browser follows the redirect to app.${BASE_DOMAIN}/, the session middleware sees no cookie, treats the user as unauthenticated, and redirects back to the OAuth provider — creating the loop.

Here's the problematic section:

// Current (broken on Safari ITP): const callbackUrl = buildRedirectUrl(baseDomain); setCookie(c, 'session', token, { httpOnly: true, secure: true, sameSite: 'None', // <-- ITP blocks this domain: `.${baseDomain}`, }); return c.redirect(callbackUrl);

The fix is to change sameSite from 'None' to 'Lax' and ensure the cookie domain matches the redirect target. Safari ITP allows Lax cookies set during top-level navigations, which is exactly what an OAuth callback is.
