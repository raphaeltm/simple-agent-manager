# Fix ULID Regex Rejection in Project Chat ACP WebSocket

## Problem

The `useProjectAgentSession` hook (`apps/web/src/hooks/useProjectAgentSession.ts:33`) uses a UUID regex to validate workspace IDs before establishing the ACP WebSocket connection:

```typescript
const UUID_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
```

All workspace IDs in the system are ULIDs (26-char Crockford Base32 strings like `01HYMKQ5W1AYDM4PQZ3T5SXQJK`), generated by `apps/api/src/lib/ulid.ts`. The UUID regex silently rejects every ULID, causing `deriveWorkspaceWsHost()` to return `''`, which means the ACP WebSocket never connects.

### Impact

1. **No real-time streaming** — Agent output doesn't stream per-token in project chat
2. **Follow-up prompts not delivered** — `handleSendFollowUp` falls to the `else` branch showing "Agent is not connected — message saved but prompt not delivered."
3. **Message rendering** — When ACP items exist exclusively, older DO-persisted history is lost (current logic shows ACP OR DO, not merged)

### Why It Works in Staging

The DO WebSocket + 3-second polling fallback still delivers persisted messages in both environments. The difference between staging and production is likely which projects/sessions are being tested — the core ACP streaming bug affects all environments equally.

## Research Findings

- **Root cause file**: `apps/web/src/hooks/useProjectAgentSession.ts:33`
- **Workspace ID generation**: `apps/api/src/lib/ulid.ts` — 26-char Crockford Base32
- **Task runner workspace creation**: `apps/api/src/durable-objects/task-runner.ts:592` — uses `ulid()`
- **UI workspace creation**: `apps/api/src/routes/workspaces.ts:517` — also uses `ulid()`
- **Message rendering**: `apps/web/src/components/chat/ProjectMessageView.tsx:632-666`
- **Follow-up handler**: `apps/web/src/components/chat/ProjectMessageView.tsx:401-458`
- **PR #270 (open)**: Contains fixes for this issue plus other unrelated changes (org installation visibility). We extract only the relevant fixes.
