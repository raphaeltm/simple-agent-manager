# Fix R2 File Upload — CORS Configuration + Postmortem

## Problem Statement

File attachments on task submissions fail with a red error indicator in the UI. Users select a file, the browser requests a presigned URL from the API (which succeeds), but the subsequent PUT to R2 via that presigned URL is blocked by the browser's CORS policy because the R2 bucket has no CORS rules configured.

Two PRs shipped this feature:
- PR #554: Full file attachment implementation (presigned URLs, task runner transfer, UI)
- PR #555: Fixed R2 credential forwarding in deployment pipeline

Despite both PRs going through the `/do` workflow, the feature still doesn't work because R2 bucket CORS was never configured. The task file's research noted "R2 CORS: Needs bucket-level CORS for direct browser uploads" and had checklist item F5 for it, but it was archived without completing the infrastructure items.

## Research Findings

### Root Cause: Missing R2 CORS Configuration
- The R2 bucket (`sam-staging-assets` / `sam-prod-assets`) has no CORS rules
- Browser XHR PUT to `{accountId}.r2.cloudflarestorage.com` is blocked by same-origin policy
- CORS must allow PUT from `app.{BASE_DOMAIN}` with Content-Type header

### What Works
- R2 credentials are set as Worker secrets (confirmed in deployment logs)
- CF_ACCOUNT_ID is set as Worker secret
- R2 bucket exists and is accessible via R2 binding
- R2_BUCKET_NAME is correctly generated by sync-wrangler-config.ts
- Presigned URL generation code is correct (AWS SDK + nodejs_compat)

### What's Missing
1. **No automated R2 CORS setup** — CORS is documented as a manual step in self-hosting.md
2. **No deployment step configures CORS** — Pulumi doesn't support R2 CORS; no script does it
3. **Task file items were not completed** — Phases F and G were never checked off before archival

### How to Fix
Add a deployment step that configures R2 bucket CORS using the S3-compatible `PutBucketCors` API. This runs after Pulumi creates the bucket and before the Worker is deployed.
