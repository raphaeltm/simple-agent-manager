# First-run admin setup wizard

- SAM Task: 01KWXSRDQTPR8B1JMAS02YP119
- SAM Idea: 01KWXQ0104CCE3KPPZ62H2MQSD
- Output branch: sam/execute-task-using-skill-01kwxs

## Problem Statement

Fresh SAM forks currently require GitHub App and OAuth credentials as deploy-time GitHub Actions secrets / Worker env vars. That makes first-run setup brittle and forces self-hosters to configure app integrations before the product can render a configuration UI.

Implement the locked first-run admin setup design: deploy with only Cloudflare credentials and trust-root generated secrets, expose a pre-auth `/setup` wizard gated by a one-time `SETUP_TOKEN` plaintext Worker var while setup is incomplete, store integration config in D1 with encrypted platform credentials, and preserve DB -> env -> unset fallback so existing staging/production continue to work unchanged.

## Research Findings

- Idea 01KWXQ0104CCE3KPPZ62H2MQSD locks the design: `SETUP_TOKEN` is delivered as a plaintext Worker `[vars]` value generated by `scripts/deploy/sync-wrangler-config.ts`; Actions output must print only a Cloudflare dashboard deep link, never the token.
- `platform_settings` already exists (`apps/api/src/db/migrations/0087_platform_settings.sql`, `apps/api/src/services/signup-approval.ts`) and provides the D1 -> env fallback pattern with `source: 'runtime' | 'environment'`.
- `platform_credentials` already exists (`apps/api/src/db/migrations/0037_platform_credentials.sql`, `apps/api/src/routes/admin-platform-credentials.ts`) and stores encrypted platform-level secrets. It needs runtime integration credential types, resolver support, and UI/API reuse.
- `apps/api/src/auth.ts` currently creates BetterAuth synchronously and reads GitHub env vars directly for `socialProviders.github` and OAuth refresh. Call sites include `routes/auth.ts`, `middleware/auth.ts`, `session-factory.ts`, `github-user-access-token.ts`, and `durable-objects/github-user-access-token-lock.ts`.
- Other direct integration config reads include `services/github-app.ts` for App ID/private key, `routes/github.ts` for app slug/install URL, `services/github-trigger-handler.ts` and `lib/secrets.ts` for webhook HMAC, plus `routes/google-auth.ts`, `routes/gcp.ts`, and `routes/project-deployment.ts` for Google OAuth credentials.
- Idea 01KWXMWBVN08E28EG8AVMQDPQT confirms Google OAuth is the low-risk non-GitHub login path. Login surfaces to update: `apps/web/src/pages/Landing.tsx`, `apps/web/src/components/trial/LoginSheet.tsx`, and `apps/web/src/pages/DeviceAuth.tsx`; client wiring is in `apps/web/src/lib/auth.ts`.
- Hono middleware scoping rule `.claude/rules/06-api-patterns.md` applies: `/setup` must be mounted pre-auth and tested through the combined app so authenticated admin wildcards cannot swallow it.
- Credential safety rules apply: `.claude/rules/28-credential-resolution-fallback-tests.md` requires behavioral tests for fallback/compare paths and atomic rate limits; `.claude/rules/41-credential-snapshot-resilience.md` requires bad stored rows not to break unrelated credential resolution; `.claude/rules/45-durable-object-concurrency-mutex.md` informs atomic critical sections.
- Migration safety `.claude/rules/31-migration-safety.md` forbids destructive migrations; add-only changes are required.
- Staging verification is mandatory via `.claude/rules/13-staging-verification.md`: after deployment, existing GitHub login users, project creation, chat, and task submission must still work.

## Acceptance Criteria

- [ ] A fresh fork can deploy with only Cloudflare credentials plus deployment-generated trust-root secrets; GitHub integration env vars are optional.
- [ ] `/setup` is unauthenticated but `SETUP_TOKEN` gated, rate-limited with an atomic primitive, live only while setup is incomplete unless `SETUP_FORCE=true`, and returns 410 Gone after completion.
