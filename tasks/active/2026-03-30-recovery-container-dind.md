# Enable Docker-in-Docker in Recovery/Fallback Devcontainer

## Problem

When a workspace's custom devcontainer build fails, the VM agent falls back to a recovery container using a default devcontainer config generated by `writeDefaultDevcontainerConfig()`. Currently, the recovery container only includes `git` and `github-cli` features — no Docker-in-Docker. An agent inside the recovery container cannot run `docker` commands or rebuild the devcontainer.

## Research Findings

### Key Files
- `packages/vm-agent/internal/bootstrap/bootstrap.go` (line 1367-1416) — `writeDefaultDevcontainerConfig()` generates the default devcontainer JSON with only `git:1` and `github-cli:1` features
- `packages/vm-agent/internal/bootstrap/bootstrap_test.go` (line 749-901) — 5 tests asserting on default config content; `TestWriteDefaultDevcontainerConfig` explicitly lists required features
- `packages/vm-agent/internal/acp/install_integration_test.go` (line 480-481) — comment about DinD exclusion is about the SAM repo's own devcontainer config, NOT the recovery container — no change needed there
- `packages/vm-agent/internal/bootstrap/bootstrap_integration_test.go` — existing integration tests use `devcontainer up` with volumes

### Implementation Approach
The change is minimal: add one line to the features map in the config JSON template, and update the test that asserts on required features.

### CI Context
- Integration tests run in `ci.yml` → `vm-agent-integration` job on `ubuntu-latest` with Docker available
- The existing integration test infrastructure already pulls devcontainer images and runs `devcontainer up`
- A dedicated DinD integration test would require `--privileged` Docker containers and 5+ minute builds — too heavyweight for this change. The unit test coverage (verifying the feature appears in the generated config) plus the existing integration tests (which exercise `runDevcontainerWithDefault()`) provide sufficient confidence.
